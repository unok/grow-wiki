# 参照（Q&A）フロー

grow-wiki に蓄積した情報を Q&A に活用するフロー。[triggers.md](triggers.md) の参照トリガーが発火した後に実行する。

## 手順

### 1. 既存ページの取得

```bash
bash .skill/scripts/list-pages.sh
```

全ページの frontmatter（title / aliases / synonyms / tags / path / last_updated）を JSON で取得。

### 2. 関連ページの特定

ユーザーの質問から**キーワード**（名詞句）を抽出し、以下の優先順位で一致を評価:

1. **title 完全一致** → 最優先
2. **aliases / synonyms 完全一致** → 次点
3. **tags が 2 個以上重なる** → 続く
4. **title / aliases に質問の語が部分含まれる** → 最後

上位 3〜5 件に絞る。候補がゼロなら「関連ページが見つかりませんでした」と伝え、通常の知識で回答する。

### 3. 本文取得

特定したページを Read で読み込む。

- メイン候補（3〜5 件）の本文ページ
- 本文内の `[[wikilink]]` のうち、回答の核に関わるものはオプションで Read（ネスト 1 段まで）
- frontmatter の `related` に載っているページも関連性が高ければ Read

**Read するページ数は 3〜5 件以内**。多すぎると context を圧迫し、要点がぼやける。

### 4. 回答合成

- ページ本文を要約・引用して回答する
- 引用箇所には出典として `[[ページ名]]` を添える
- 回答の末尾に「参照したページ」のリストを出す:

  ```
  📚 参照: [[ページA]], [[ページB]], [[ページC]]
  ```

- 本文の `last_updated` が古い（例: 1 年以上前）場合は「vault の情報は X 時点のものです」と断る
- 本文の内容と最新の知識（会話中の発言や公開情報）が食い違う場合、両方を提示して判断を求める

### 5. 更新提案

回答の過程で新情報や訂正に気づいたら、**書き込み系の ingest フロー**に切り替えて「grow-wiki の [[〇〇]] を更新しますか？」と提案する。承認後に [update-logic](update-logic.md) に従って追記/再生成。

## キーワード抽出のヒント

- ユーザー発話の名詞・固有名詞を優先
- 日本語は形態素解析せず「まとまった語」として扱って OK
- 英略語・製品名（`React`, `DDD`, `Event Sourcing` 等）は一語として扱う
- 動詞・形容詞は抽出しない
- 一般語（「設計」「実装」「方法」「理由」）は vault の tag と一致していても除外

## 参照結果の提示フォーマット

### 複数ページを参照した場合

```
[回答本文。適宜 [[ページ名]] で参照を示す]

📚 参照:
- [[ページA]] — 主要参照
- [[ページB]] — 補足
- [[ページC]] — 関連
```

### 情報が不足している時

```
grow-wiki には [[〇〇]] に関連する断片しかありません（直接的な解答はなし）。
以下の関連ページが近そうです:
- [[ページA]]
- [[ページB]]

一般的な知識で回答しますか？ それとも grow-wiki に新規ページとして記録しますか？
```

### 矛盾・陳腐化を検出した時

```
vault の [[〇〇]]（last_updated: YYYY-MM-DD）には「...」とありますが、
最新の情報では「...」となっています。

どちらを信頼しますか？ 更新する場合は [[〇〇]] を最新情報で上書きします。
```

## 過剰発火の回避

- 同一会話内で一度「no」と答えた話題は再提案しない
- 1 会話で自動発火は 2〜3 回までを目安（連続発火は抑制）
- マッチ数が 5 件を超える時は具体性不足 → 提案せず聞き返す
- 参照したページは会話の context に一度入れば良い。同じ話題で複数回 Read しない
